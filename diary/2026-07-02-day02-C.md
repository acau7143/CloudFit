## 왜 이 기술/방법을 선택했는가
| 항목 | 선택한 것 | 대안 | 선택 이유 |
|------|-----------|------|-----------|
| SSH 접속 | SSH 키 방식 (MobaXterm) | gcloud SSH | MobaXterm이 편하고 별도 창 안 열어도 됨 |
| 인증 방식 | VM 서비스 계정 자동 인증 (Compute Engine 메타데이터) | ADC (gcloud auth) | VM 안에서는 서비스 계정이 자동으로 인증됨 |
| 가상환경 | venv | 시스템 pip | Debian 12부터 시스템 pip 직접 설치 막혀있음 |
| 권한 부여 | IAM 역할 (`roles/monitoring.viewer`) | 서비스 계정 키 | 키 발급이 조직 정책으로 막혀있고, IAM 역할이 더 안전 |

## 오늘 처음 안 사실
| 몰랐던 것 | 알게 된 것 | 왜 그렇게 되는지 |
|-----------|-----------|----------------|
| /tmp 폴더가 날아가는 이유 | VM 재시작하면 /tmp는 초기화됨 | /tmp는 임시 저장소라서 영구 저장이 안 됨 |
| CPU가 100% 넘는 이유 | GCP는 vCPU 수 기준이라 멀티코어면 100% 초과 가능 | vCPU 수로 나눠야 정확한 % 나옴 |
| VM 안에서 인증이 자동으로 되는 이유 | GCP VM은 Compute Engine 메타데이터 서버로 자동 인증 | VM 안에 서비스 계정이 자동으로 연결되어 있음 |
| Monitoring API가 따로 있는 이유 | GCP는 서비스마다 API를 별도로 활성화해야 함 | 보안/비용 관리를 위해 사용하는 API만 켜두는 구조 |
